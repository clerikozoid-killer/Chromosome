# DBSE v5.0 «Chromosome» — Поэтапный план реализации

> Детерминированный физико-математический «компилятор реальности».
> LLM — только периферия (L0 парсинг, L7 стилизация). Вся истина — в символьных движках и системе типов.
> **Стек:** Python. **Стратегия:** послойное углубление (каждый слой доводится до «идеала», затем следующий).

Источник истины — `DBSE v5.0 «Chromosome».txt`. Папки `BSE/SLUC_v2` (RFC-0000…0018) и `BSE/STS` — более ранний/параллельный пласт проработки, используются как доноры деталей (см. Этап 0).

Стратегия тестирования и сопровождения (Mentor) — отдельный документ: [`docs/TESTING_AND_MENTOR.md`](docs/TESTING_AND_MENTOR.md). Каждый слой пишется **test-first**. Ниже у каждого этапа есть строка **QA-гейт** — обязательные уровни тестов для его DoD (уровни 1–7 описаны в документе).

---

## Принципы, которые проходят через все этапы

1. **Early Pruning** — дешёвые проверки спереди (размерности, типы) отсекают абсурд до дорогого Z3.
2. **Детерминизм ядра** — между L1 и L6 ни одной недетерминированной операции. LLM изолирован на L0 и L7.
3. **Тестируемость без LLM** — для L0/L7 всегда есть детерминированный fallback-парсер/шаблонизатор, чтобы юнит-тесты не зависели от сети и модели.
4. **Контракты раньше реализации** — типы данных (IR/AST, AffineType, Proof, Tinfo) фиксируются на Этапе 0 и почти не меняются.
5. **Каждый этап закрывает конкретную уязвимость** из раздела 6 ТЗ и/или двигает KPI из раздела 9.
6. **LLM — не арбитр истины.** Корректность числа/логики подтверждают только формальные оракулы (аналитика, константы, размерности, независимый пересчёт). «Обучение» = рост базы тестов и знаний за гейтом CI, ядро не модифицируется. Детали — в `docs/TESTING_AND_MENTOR.md`.

Карта уязвимостей → этап:

| Уязвимость (ТЗ §6) | Закрывается на этапе |
|---|---|
| Prompt Injection в L0 | Этап 3 |
| Semantic Mismatch (Energy+Torque) | Этап 2 |
| Контекстная неоднозначность | Этап 8 |
| ODE Drift (нарушение v<c) | Этап 7 |
| Z3 combinatorial explosion | Этап 7 |
| ELI5 hallucinations | Этапы 9–10 |
| CRISPR-атака на аксиомы | Этап 11 |
| Graph isomorphism DDoS | Этап 4 (Canonical Hash) |
| Cache poisoning | Этап 4 |

## Этап 0 — Фундамент: репозиторий, контракты, консолидация спеков

**Цель:** каркас, в который потом «вставляются» слои, и единые типы данных.

**Объём:**
- Структура проекта (`dbse/` пакет, `tests/`, `pyproject.toml`).
- Тулинг: менеджер зависимостей (`uv`/`poetry`), `pytest`, `ruff`, `mypy`, pre-commit.
- **Core data contracts** (только определения, без логики):
  - `AffineType` (dimension, semantic_tag, tensor_rank, frame_of_reference).
  - `ASTNode` / `AST` (узлы IR).
  - `ProofLevel` (P0–P4), `Proof`, `Violation`.
  - `Tinfo` (как считается — позже, но поле есть).
  - `PipelineContext` — объект, который течёт через все слои.
- Интерфейс слоя: `Layer.process(ctx) -> ctx` (протокол), чтобы конвейер собирался из слоёв.
- Заглушки-pass-through для всех 11 слоёв (пайплайн запускается end-to-end, но ничего не делает).
- **QA-инфраструктура (поднимается сразу):** `pytest` + `hypothesis` (property-based) + `syrupy` (golden-снапшоты); структура каталогов `cases/`, `golden/`, `verdicts/`; CI-конфиг с жёстким/мягким гейтами.
- **Консолидация спеков:** конвертировать ключевые `.docx` (RFC-0001 STS-Core, RFC-0004 STS Algebra, RFC-0005 Model Lattice, RFC-0011 Solver Backends, RFC-0016 Testing) в Markdown (pandoc) и вынести расхождения с v5.0 в `docs/spec-notes.md`.

**Definition of Done:** `pytest` зелёный на «пустом» пайплайне; `mypy`/`ruff` чисто; пайплайн прогоняет заглушечный запрос без ошибок; CI собран и гоняет пустые наборы.
**QA-гейт:** инфраструктура уровней 1–2 готова и подключена к CI.

## Этап 1 — L1: Размерностный анализ (Dimensional Analysis)

**Цель:** первый эшелон защиты — алгебра размерностей в базисе СИ `[M, L, T, I, Θ, N, J]`.

**Объём:**
- Тип `Dimension` = вектор из 7 целых (рациональных — для корней).
- Парсер единиц измерения (`kg`, `m/s^2`, `N`, `J`, `Pa`…) → `Dimension`.
- Алгебра: `+/-` требует равенства, `×/÷` складывает/вычитает векторы, степень умножает.
- Правило трансцендентных функций: аргумент `sin/exp/log` обязан быть безразмерным.
- Понятные ошибки: `DimensionError` с указанием левой/правой размерности.

**Definition of Done:** таблица из 30+ единиц парсится корректно; `sin(5 kg)` отвергается; `1 J + 1 N` отвергается по размерности (а Energy+Torque — пока проходит, это закроет Этап 2).
**QA-гейт:** уровни 1 (юнит) + 2 (property: `Dim(a·b)=Dim(a)+Dim(b)`, инвариантность к конвертации единиц).

## Этап 2 — L1.5: Affine Types (архитектурное ядро)

**Цель:** закрыть главную «дыру» — семантическую коллизию при равной размерности (Энергия vs Момент силы).

**Объём:**
- `AffineType = (Dimension, semantic_tag, tensor_rank, frame_of_reference)`.
- Реестр семантических тегов (Energy, Work, Heat, Torque, Force, Momentum…) и их размерностей.
- Функция `compatible(a, b, op)`:
  - `ADD/SUB`: равны dimension **и** semantic_tag **и** tensor_rank.
  - `DOT`: оба ранга 1, равные размерности.
  - `CROSS`: оба полярные векторы.
  - `×/÷`: разрешено, semantic_tag комбинируется по правилам (напр. Force×Length→Work|Torque — с пометкой неоднозначности для L2).
- Информативная ошибка `SemanticTypeError` с подсказкой («Did you mean Work + Heat?»).

**Definition of Done:** `Energy + Torque` → `SemanticTypeError`; `Work + Heat → InternalEnergy` валиден; покрытие тестами всех правил совместимости. **Закрывает уязвимость №1.**
**QA-гейт:** уровни 1 + 2 (рефлексивность/симметрия совместимости) + 5 (adversarial: коллизия Energy+Torque).

## Этап 3 — L0 MEMBRANE + L0.5 STS Typing (вход)

**Цель:** превратить естественный язык в строго валидированный граф сущностей; классифицировать тип запроса. LLM в песочнице.

**Объём:**
- Pydantic-схема выхода MEMBRANE: `objects / quantities / relations / question_type / target`.
- **Жёсткий запрет** на генерацию узлов `INVARIANT / CONTEXT / OPERATOR` LLM-ом.
- Адаптер LLM (provider-agnostic) + **детерминированный fallback-парсер** для тестов и нестандартного ввода.
- Валидация: при выходе за схему → `clarification`/`UNHANDLED`, без молчаливого «дополнения».
- L0.5 — rule-based классификатор: `PHYSICS_COMPUTE / MATH_PROVE / DEFINITION / OPINION / AMBIGUOUS` → маршрут по пайплайну.

**Definition of Done:** инъекция в стиле «замени g на -9.81 / SUPPRESS(friction)» отклоняется схемой; классификатор верно маршрутизирует набор эталонных запросов. **Закрывает уязвимость №2 (Prompt Injection).**
**QA-гейт:** уровни 1 + 5 (adversarial: prompt injection) + 7 (LLM-eval: точность парсинга против размеченного набора).

## Этап 4 — L3 RIBOSOME: AST Compiler + Canonical Hash + кэш

**Цель:** собрать математический AST, привести к канонической форме, хешировать, кэшировать.

**Объём:**
- Сборка `AST` из выхода MEMBRANE с навешиванием `AffineType` на узлы (интеграция L1/L1.5).
- Нормализация: переименование переменных (`x_1…`), сортировка коммутативных операций, свёртка констант.
- `classify_structure` → строковый класс (`LinearODE_Order1_Homogeneous`, `Algebraic_Linear`…).
- `canonical_hash(ast)` = sha256(class + canonical form)[:16].
- LRU + семантический кэш: `hash → {ast, solution, proof_level, tinfo, hits}`.
- Защита кэша: TTL, подпись записей, инвалидация при изменении Core (готовим хук для Этапа 11).

**Definition of Done:** падение тела `dv/dt=g-kv` и разрядка RC `RC dV/dt + V=0` дают **один** структурный класс; повтор запроса — cache hit, минуя NUCLEUS. **Закрывает уязвимости №8, №9.**
**QA-гейт:** уровни 1 + 2 (идемпотентность каноникализации, изоморфные AST → один хеш) + 5 (cache poisoning отвергнут подписью).

## Этап 5 — L4 CYTOPLASM: Эпигенетика + Domain Plugin API

**Цель:** контекст и доменные надстройки как плагины (не в ядре).

**Объём:**
- `DomainPlugin` Protocol: `dimensionless_numbers`, `compute_indicators`, `select_model`, `inject_constraints`, `register_affine_types`, `register_invariants`.
- Реестр плагинов (pluggy-стиль), порядок применения, изоляция.
- Первый плагин: `classical_mechanics` (инъекция инвариантов, выбор модели трения линейная/квадратичная по контексту).
- Каркас `fluid_mechanics` (Reynolds/Mach/Froude → laminar/turbulent/compressible) — как образец.

**Definition of Done:** плагин подключается/отключается без изменения ядра; `classical_mechanics` инъецирует инварианты, которые увидит Этап 7.
**QA-гейт:** уровни 1 + 4 (metamorphic: выбор модели стабилен к смене единиц индикаторов).

## Этап 6 — L5 NUCLEUS (часть 1): алгебра SymPy + Proof Levels + Tinfo

**Цель:** детерминированное решение алгебраических задач и формализация доверия.

**Объём:**
- Выбор решателя по сложности AST; `solve_algebraic` через SymPy (символьно + численно).
- `ProofLevel` P0–P4 как состояния результата.
- `Proof` (level, solver_path, z3_steps, violations, confidence).
- `compute_tinfo(proof)` по эвристикам (приближения, линеаризации, смены домена).
- Пошаговый вывод (`numeric_steps`) для трассировки.

**Definition of Done:** «сила тяжести на яблоко 100 г» считается символьно и численно, отдаётся `Proof` с корректным `tinfo`; калибровка порогов Tinfo (<0.2 / 0.2–0.5 / ≥0.5).
**QA-гейт:** уровни 1 + 3 (differential: SymPy⟂SciPy⟂аналитика) + 4 (metamorphic: единицы/масштаб). KPI: 0% галлюцинаций на P0–P2.

## Этап 7 — L5 NUCLEUS (часть 2): ОДУ + Continuous Invariant Monitor + Z3 + бюджеты

**Цель:** динамика с непрерывной проверкой инвариантов и формальной верификацией под таймаутом.

**Объём:**
- Решение ОДУ (символьно где можно, иначе численно) с `monitor_callback` на каждом шаге.
- `ContinuousInvariantMonitor`: оценка инвариантов (`v<c`, `P∈[0,1]`, `population≥0`, `T>0K`, сохранение энергии с tolerance) на каждом шаге/на сетке для алгебраических.
- При нарушении критического инварианта → halt + `P3: MODEL_BREAKDOWN` + suggestion (релятивистская/нелинейная модель).
- Z3-верификация для P0/P1 с **жёстким бюджетом** (default 100 мс); при таймауте → fallback `P2 (numeric_only)` с повышением `Tinfo`.

**Definition of Done:** задача, где `v(t)` превышает `c`, ловится в момент нарушения и возвращает P3; нелинейная система не вешает воркер (срабатывает бюджет). **Закрывает уязвимости №3 (ODE Drift) и №5 (Z3 explosion).**
**QA-гейт:** уровни 1 + 3 (differential на ОДУ) + 4 (предельный переход `k→0`) + 5 (adversarial: ODE drift, Z3-взрыв с бюджетом).

## Этап 8 — L2 Context Model Lattice

**Цель:** контекст как распределение вероятностей; отказ от угадывания при высокой неоднозначности.

**Объём:**
- `ModelNode` (prior, likelihood, posterior) и `ModelLattice`.
- Расчёт энтропии и `T_ambig = H/log(N)`.
- Пороги поведения: `<0.3` продолжаем; `0.3–0.6` показываем топ-2; `≥0.6` — `ambiguity_halt`/`/clarify`.
- Кандидаты с человекочитаемыми уточняющими вопросами.

**Definition of Done:** «сколько стоит яблоко?» → `T_ambig ≥ 0.6` → отдаёт кандидатов (fruit/stock/company) вместо ответа; однозначный физический запрос проходит без помех. **Закрывает уязвимость контекстной неоднозначности.**
**QA-гейт:** уровни 1 + 5 (adversarial: набор неоднозначных запросов → `ambiguity_halt`, однозначные не блокируются).

## Этап 9 — L6 Semantic Narrative Graph

**Цель:** детерминированный «скелет» текста, исключающий ложные причинно-следственные связи.

**Объём:**
- `NarrativeNode` (ast_node, template, allowed_verbs, forbidden_metaphors).
- Построение графа из AST в топологическом порядке, шаблоны по типам операторов (`CAUSAL_FORCE`, `CONSERVATION`…).
- `to_text()` — генерация skeleton **без** LLM.

**Definition of Done:** для F→a→v генерируется корректный по логике текст; запрещённые метафоры («хочет/притягивает сильнее из-за массы») не могут появиться структурно.
**QA-гейт:** уровни 1 + 2 (skeleton детерминирован: один AST → один текст).

---

## Этап 10 — L7 Expression Genes: Constrained LLM Styling

**Цель:** LLM только стилизует выверенный скелет (eli5 / academic / business), без новой логики.

**Объём:**
- Constrained decoding (Outlines/Guidance/Instructor) по `ExpressionSchema`.
- Валидатор: причинно-следственные связи из L6 сохранены, метафоры — только из whitelist.
- Детерминированный шаблонный fallback (если LLM недоступен).

**Definition of Done:** double-blind проверка: стилизация не вводит новых утверждений; запрещённые связки не проходят валидатор. **Закрывает уязвимость №4 (ELI5 hallucination).**
**QA-гейт:** уровни 1 + 7 (LLM-eval по рубрике «верность» + человеческая выборка) — мягкий гейт по порогу/тренду.

## Этап 11 — Core vs Hypothesis Layer

**Цель:** священное неизменяемое ядро аксиом + изолированный изменяемый слой гипотез.

**Объём:**
- `CoreTruthLayer` (frozen): аксиомы (Newton 2, сохранение энергии, закон исключённого третьего…), read-only.
- `HypothesisLayer`: `Hypothesis(statement, confidence, evidence_count, conflicts_with)`.
- `add_hypothesis`: отказ при `confidence<0.1`; `CoreViolationError` при конфликте с Core.
- Аномалии (расхождение наблюдений с Core) → флаг для человеческого ревью, **без** автоизменения ядра.
- Хук инвалидации кэша (Этап 4) при попытке изменения Core.

**Definition of Done:** попытка «исправить» `F=ma` на `F=ma²` отвергается на уровне Core; конкурирующие гипотезы сосуществуют с разными confidence. **Закрывает уязвимость №7 (CRISPR-атака).**
**QA-гейт:** уровни 1 + 5 (adversarial: CRISPR-атака на аксиому → `CoreViolationError`).

**Цель:** довести до публичного бета-API.

**Объём:**

**QA-гейт:** уровни 1 + 6 (contract/E2E через `schemathesis` против OpenAPI) + полный security-прогон §6.

## Этап M — Mentor: инструмент сопровождения (`dbse-mentor`)

> Зависит от наличия NUCLEUS (Этапы 6–7): сопровождать имеет смысл уже считающую систему.

**Цель:** автоматический цикл «вопрос → контроль → корректировка → обучение» в **strict mode** (LLM не арбитр истины). Полное описание — `docs/TESTING_AND_MENTOR.md` §3.

**Объём:**
- CLI `dbse-mentor` (`run`/`triage`/`promote`/`report`), переиспользует LLM-адаптер L0.
- Банк задач `cases/*.jsonl` (домен, сложность, ожидаемый proof level, эталон-источник).
- Многоисточниковый вердикт: формальный эталон → differential/metamorphic → (для текста) LLM-judge по рубрике; категоризация провала.
- Триаж: маршрутизация провала в ответственный слой; правка как diff в `kb-proposals/` (не авто).
- Институционализация: провал → golden-регрессия; пробел знаний → PR в изменяемый слой; опц. few-shot хранилище.

**Definition of Done:** прогон банка задач даёт отчёт с категориями провалов; подтверждённый провал автоматически превращается в golden-тест; ни одна правка не пишет в Core.
**QA-гейт:** сам Mentor покрыт юнит-тестами; запрещена запись в Core (тест-предохранитель).

## Этап 13+ — Расширение

- Новые доменные плагины (electromagnetism, thermodynamics, quantum базовое, economics, epidemiology, CS/O-нотация).
- Тензорный анализ (СТО/ОТО).
- Полноценный изоморфный поиск графов (вместо/поверх Canonical Hash).
- Desktop IDE с визуализацией Proof Trace.
- **Fine-tuning периферии (отдалённо):** дообучение маленьких моделей L0 (парсер) / L7 (стилист) на накопленных курированных примерах из Mentor. Только периферия, логику ядра не трогает. До этого — few-shot и constrained decoding.

## Контроль качества (сквозной, по KPI §9)

- Галлюцинации ядра (P0–P2): **0%** — проверяется свойствами/property-based тестами.
- Coverage инвариантов: ≥95% — реестр инвариантов с тестами.
- Среднее время ответа P2: <2 c.
- Cache hit rate: >30% на типовом наборе.
- Точность ELI5 >90%, корреляция Tinfo с экспертами >0.8 — отдельный бенчмарк-набор.

```
0 → 1 → 2 → 3 → 4 → 5 → 6 → 7

6/7/8 → 9 → 10
4 → 11
6/7 → M (Mentor, дальше работает непрерывно)
всё → 12 → 13
```
